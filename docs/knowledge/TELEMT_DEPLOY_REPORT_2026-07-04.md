## 7. Issues Encountered

### 7.1 `angie/angie:latest` Docker image does not exist

**Severity:** Blocker
**Symptom:** `pull access denied for angie/angie, repository does not exist or may require 'docker login'`
**Root cause:** The `angie/angie` image is not published on Docker Hub under that name. Angie's official Docker image is not available through standard registries.
**Fix applied:** Replaced with `socheatsok78/angie:1.11.7-ubuntu` (community-maintained Angie image on Docker Hub).
**Impact on repo:** `docker-compose.yml` files in `infra/exit/`, `infra/mgmt/`, and `infra/landing/` all reference `angie/angie:latest`. All three need to be updated.
**Suggested fix for repo:** Either:
- Use `socheatsok78/angie:1.11.7-ubuntu` (verified working)
- Build a custom Angie image from official binaries
- Check if Angie has an official GHCR image

### 7.2 Xray config mount path mismatch

**Severity:** Blocker
**Symptom:** Xray container starts but reads default configs from `/usr/local/etc/xray/00_log.json`, `01_api.json`, etc. instead of the mounted `config.json`. Our VLESS-Reality config is ignored.
**Root cause:** `docker-compose.yml` mounts the config to `/etc/xray/config.json`, but the `ghcr.io/xtls/xray-core` image uses `/usr/local/etc/xray/` as the config directory. The image has built-in default JSON configs at that path.
**Fix applied:** Changed volume mount to `./xray-config.json:/usr/local/etc/xray/config.json:ro`. Now Xray reads our config AFTER the default configs, appending our inbound/outbound.
**Impact on repo:** `infra/exit/docker-compose.yml` and `infra/entry/docker-compose.yml` both have wrong mount paths.
**Suggested fix for repo:** Update both compose files:
```yaml
volumes:
  - ./xray-config.json:/usr/local/etc/xray/config.json:ro
```

### 7.3 telemt config.toml not loaded (wrong working directory)

**Severity:** Blocker
**Symptom:** telemt logs show `TLS domain: petrovich.ru` (built-in default) instead of `www.microsoft.com` from our config.
**Root cause:** telemt's Docker image has `WorkingDir: /app` and `Cmd: [config.toml]`. This means it looks for `config.toml` in `/app/config.toml`. But `docker-compose.yml` mounts the config directory to `/etc/telemt/`, so the file is at `/etc/telemt/config.toml` — telemt can't find it.
**Fix applied:** Added `command: ["/etc/telemt/config.toml"]` to the telemt service in `docker-compose.yml`.
**Impact on repo:** `infra/exit/docker-compose.yml` telemt service missing `command` directive.
**Suggested fix for repo:** Add to telemt service:
```yaml
command: ["/etc/telemt/config.toml"]
```

### 7.4 `config_strict = true` rejects `access.user_data_quota_bytes`

**Severity:** Blocker
**Symptom:** telemt crashes with `Config error: unknown config keys are not allowed when general.config_strict=true: access.user_data_quota_bytes`
**Root cause:** The `config.toml.template` includes `[access.user_data_quota_bytes]` section with `free_user = 1073741824`. telemt 3.4.22 does not recognize this key when `config_strict = true`.
**Fix applied:** Set `config_strict = false` in `config.toml` for testing. telemt now logs a warning (`Unknown config key ignored`) but starts successfully.
**Impact on repo:** `infra/exit/config.toml.template` includes a key not supported by telemt 3.4.22.
**Suggested fix for repo:** Either:
- Remove `access.user_data_quota_bytes` from the template
- Set `config_strict = false` by default
- Document which telemt versions support this key

### 7.5 Xray cannot bind port 443 (permission denied)

**Severity:** Blocker
**Symptom:** `failed to listen TCP on 443 > bind: permission denied`
**Root cause:** Docker container with `cap_drop: [ALL]` cannot bind ports below 1024. The compose file has `cap_add: [NET_BIND_SERVICE]` which should allow it, but it doesn't work with the `socheatsok78/angie` image or possibly the xray-core image.
**Fix applied:** Added `user: root` to xray services in docker-compose.yml. Removed `read_only: true` for testing.
**Impact on repo:** Both entry and exit compose files have this issue.
**Suggested fix for repo:** The `cap_add: [NET_BIND_SERVICE]` should work in theory. Needs investigation — may be an image-specific issue. For production, `user: root` is not ideal.

### 7.6 Angie (telemt-mask) crashes on chown/setgid

**Severity:** Blocker
**Symptom:** `chown("/var/cache/angie/client_temp", 999) failed (1: Operation not permitted)` and `setgid(999) failed (1: Operation not permitted)`
**Root cause:** Container is `read_only: true` with `cap_drop: [ALL]`. The `socheatsok78/angie` image's entrypoint tries to chown cache directories and setgid to the angie user (UID 999), but lacks `CHOWN` and `SETGID` capabilities.
**Fix applied:** Added `CHOWN`, `SETGID`, `SETUID` to `cap_add`. Removed `read_only: true`.
**Impact on repo:** All three compose files using Angie have this issue.
**Suggested fix for repo:** Either:
- Add `CHOWN`, `SETGID`, `SETUID` to `cap_add`
- Use an Angie image that doesn't require these (distroless or rootless)
- Pre-create cache directories with correct permissions

### 7.7 `INFRA_DIR` path bug in deploy scripts

**Severity:** Blocker
**Symptom:** `deploy-exit.sh: line 66: /opt/telemt-mgmt/lib/common.sh: No such file or directory`
**Root cause:** Deploy scripts resolve `INFRA_DIR` as `$SCRIPT_DIR/../..` (two levels up from `infra/exit/` = repo root). Then source `$INFRA_DIR/lib/common.sh` = `/opt/telemt-mgmt/lib/common.sh`. But `common.sh` is at `/opt/telemt-mgmt/infra/lib/common.sh`.
**Fix applied:** Created symlink: `ln -sf infra/lib /opt/telemt-mgmt/lib`
**Impact on repo:** All deploy scripts (`deploy-exit.sh`, `deploy-entry.sh`, etc.) have this bug.
**Suggested fix for repo:** Change `INFRA_DIR` from `$SCRIPT_DIR/../..` to `$SCRIPT_DIR/..` (one level up = `infra/`), then source `$INFRA_DIR/lib/common.sh`.

### 7.8 telemt `proxy_protocol = true` blocks direct connections

**Severity:** Functional (expected behavior, not a bug)
**Symptom:** Direct connections to telemt :8443 are rejected with `Invalid proxy protocol header`. TLS handshake fails.
**Root cause:** telemt is configured with `proxy_protocol = true` — it expects a PROXYv1 header on every incoming connection. This is correct for the double-hop architecture (xray-exit sends PROXYv1), but blocks direct client connections for testing.
**Fix applied:** Set `proxy_protocol = false` and `mask_proxy_protocol = 0` in `config.toml` for testing. Direct connections now work.
**Note:** This is NOT a repo bug — it's correct behavior. For production with double-hop, `proxy_protocol = true` is correct. For direct testing, it must be disabled.

### 7.9 telemt `tls_emulation = true` fetch fails

**Severity:** Functional (expected for test setup)
**Symptom:** telemt logs warn: `TLS emulation initial fetch failed domain=www.microsoft.com error=upstream route connect failed for www.microsoft.com:8080: Connection timeout`
**Root cause:** telemt's `tls_emulation` feature tries to fetch a real ServerHello from `mask_host:mask_port` (www.microsoft.com:8080). In third-party mode, `mask_port = 8080`, but `www.microsoft.com` doesn't serve on port 8080. The mask_host should be the exit server's own Angie on :8080, but `mask_host` is set to `www.microsoft.com` (the TLS domain), not `localhost`.
**Fix applied:** Set `tls_emulation = false` for testing. telemt uses its built-in TLS fingerprint instead.
**Impact on repo:** The `mask_host` / `mask_port` logic in `config.toml.template` may be incorrect for third-party mode. When `mask_host = www.microsoft.com` and `mask_port = 8080`, telemt tries to connect to `www.microsoft.com:8080` — which doesn't exist. The intent was for Angie on :8080 to serve the mask page, but telemt's `tls_emulation` fetches from `mask_host:mask_port`, not from localhost.
**Suggested fix for repo:** Clarify the relationship between `mask_host`, `mask_port`, and `tls_emulation`. In third-party mode, `tls_emulation` should either:
- Fetch from the real third-party domain on :443 (set `mask_port = 443`)
- Be disabled (set `tls_emulation = false`)
- Use localhost as mask_host when Angie serves the mask

### 7.10 Xray `xray x25519` / `xray uuid` command format

**Severity:** Minor (in deploy scripts)
**Symptom:** `docker run --rm ghcr.io/xtls/xray-core:latest xray x25519` → `xray xray: unknown command`
**Root cause:** The xray-core Docker image's entrypoint is already set to `xray`, so passing `xray x25519` becomes `xray xray x25519` (double command). The deploy scripts use `docker run --rm ghcr.io/xtls/xray-core:latest xray x25519` but should use `docker run --rm --entrypoint /usr/local/bin/xray ghcr.io/xtls/xray-core:latest x25519` or simply `docker run --rm ghcr.io/xtls/xray-core:latest x25519` (without the extra `xray` prefix).
**Fix applied:** Used `--entrypoint /usr/local/bin/xray` for key generation.
**Impact on repo:** `deploy-exit.sh` lines 343, 387 and `deploy-entry.sh` lines 164, 232.
**Suggested fix for repo:** Change `docker run --rm ghcr.io/xtls/xray-core:latest xray x25519` to `docker run --rm ghcr.io/xtls/xray-core:latest x25519` (remove the extra `xray` prefix, since the entrypoint is already `xray`).

### 7.11 `tg://proxy` link does not work through entry (VLESS-Reality)

**Severity:** Architectural gap
**Symptom:** telemt API returns `tg://proxy?server=168.119.59.155&port=8443&secret=ee...` — this points to the EXIT server directly, not the entry server.
**Root cause:** In ARCH-001@0.2.0 (ADR-009, encrypted S2), the entry server speaks VLESS-Reality on :443, not MTProto. A `tg://proxy` link makes Telegram connect with MTProto protocol, but the entry server expects VLESS-Reality. The two protocols are incompatible at the wire level.
**Impact:** The project's `link.py` (`build_proxy_link()`) still generates `tg://proxy?server=<entry_domain>&port=443&secret=<secret>` links. These links will NOT work with the 0.2.0 encrypted S2 architecture — Telegram clients cannot speak VLESS-Reality.
**For double-hop testing:** A VLESS client (xray/v2ray) is required. The VLESS link format is:
```
vless://<UUID>@<entry_domain>:443?encryption=none&security=reality&sni=<SNI>&fp=firefox&pbk=<PUB_KEY>&sid=<SHORT_ID>&type=tcp&flow=xtls-rprx-vision
```
**For direct testing:** The `tg://proxy` link to the exit server (port 8443) works, bypassing the entry server entirely.
**Suggested fix for repo:** This is a fundamental architecture question:
- If double-hop with encrypted S2 is the target: `link.py` should generate VLESS links, not `tg://proxy` links. But VLESS links require an xray/v2ray client, not Telegram's built-in proxy support.
- If `tg://proxy` links are required (Telegram built-in): the entry server must be a transparent TCP forwarder (freedom outbound, not VLESS-Reality). This means no encrypted S2 — MTProto patterns are visible on the RU→EU segment.
- The 0.1.x architecture (freedom outbound on entry) supported `tg://proxy` links. The 0.2.0 architecture (VLESS-Reality outbound) broke this.
- This is the most significant finding from the test deployment.

### 7.12 telemt API auth header format

**Severity:** Minor
**Symptom:** `Authorization: Bearer <token>` returns 401 Unauthorized.
**Root cause:** telemt API expects the raw token as the Authorization header value, without the `Bearer ` prefix.
**Fix applied:** Use `Authorization: <token>` (no Bearer prefix).
**Impact on repo:** `TelemtClient` in `telemt_proxy/client.py` sends `Authorization: <auth_header>` — this is correct. The `.env.example` and deploy script comments should document this clearly.
**Note:** This is a documentation issue, not a code bug. The code is correct.

### For the architectural gap (#11)

The 0.2.0 encrypted S2 architecture (ADR-009) fundamentally changes how
clients connect. Options:

- **Option A:** Keep encrypted S2, generate VLESS links. Clients need
  xray/v2ray. No Telegram built-in proxy support.
- **Option B:** Revert entry to `freedom` outbound (0.1.x). tg://proxy
  links work with Telegram built-in. No encrypted S2.
- **Option C:** Support both modes. Entry can be deployed with either
  `freedom` (tg://proxy) or `vless` (VLESS client) outbound. Operator
