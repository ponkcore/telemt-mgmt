# ADR-002: Admin Panel Authentication via JWT

## Context

ARCH-001@0.1.0 §3 C3 needs an authentication mechanism for the admin panel. The PO chose username/password + JWT tokens to support multiple admin accounts (decision fork #2). The PRD Non-Goals exclude user-facing auth, but the operator's admin panel requires protection.

## Decision

We will implement JWT-based authentication for the admin API:
- `POST /api/auth/login` accepts `{username, password}`, validates against `admin_users` table (bcrypt-hashed passwords), returns a JWT access token (HS256, 24h expiry).
- All other admin endpoints require `Authorization: Bearer <token>` header.
- Admin users are created by the deploy script (`deploy-mgmt.sh` prompts for initial admin username/password) and can be managed via a CLI command.
- JWT secret key is stored in env var `JWT_SECRET_KEY` (auto-generated by deploy script if not provided).
- Rate limiting on `/api/auth/login`: 5 failed attempts per IP per minute (429 Too Many Requests).

## Consequences

- **Positive:** Supports multiple admin accounts from day one. No shared secret to distribute.
- **Positive:** JWT is stateless — no server-side session storage needed.
- **Negative / cost:** More complex than a simple API key. Requires `admin_users` table, password hashing, token validation middleware.
- **Negative / cost:** JWT tokens are not revocable without a blacklist. Mitigation: short expiry (24h) + frontend auto-refresh.
- **Follow-ups:** Frontend must handle 401 responses by redirecting to login page. `deploy-mgmt.sh` must prompt for initial admin credentials.

## Alternatives considered

- **Simple API key (shared secret)** — PO rejected; wants multiple admin support.
- **OAuth2 via Telegram Login Widget** — considered; PO chose password-based for simplicity.
